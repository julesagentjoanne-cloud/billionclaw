# startupClaw - Research Inputs

## Purpose
Define the research inputs startupClaw should use to expand founder input into stronger startup and GTM hypotheses.

## Simple overview
startupClaw should not rely only on what the founder types.
It should combine founder input with external evidence sources to suggest:
- audience options
- pain points
- alternatives / competitors
- willingness-to-pay clues
- what still needs validation

## Core research inputs

### 1. Founder input
Use for:
- startup description
- problem solved
- initial audience guess
- initial competitor/alternative guess

### 2. Forums and communities
Examples:
- Reddit
- niche forums
- Discord/Slack communities

Use for:
- raw pain language
- repeated frustrations
- workarounds
- trigger moments

### 3. Review sites
Examples:
- G2
- Capterra
- Trustpilot
- app marketplaces

Use for:
- dissatisfaction with current tools
- missing features
- switching reasons
- expectation vs reality
- competitor gaps
- willingness-to-pay frustration signals

### 4. Competitor pages
Use for:
- category framing
- feature expectations
- positioning patterns
- proof expectations

### 5. Search behavior / query patterns
Use for:
- problem framing
- comparison intent
- demand language
- solution-seeking behavior

### 6. Customer interviews
Use for:
- importance validation
- urgency validation
- buying-process insight
- direct willingness-to-pay insight

### 7. Product / funnel / CRM data
Use for:
- real behavior
- conversion leaks
- activation/retention pain
- channel performance

### 8. Job descriptions / vacancies
Use for:
- evidence of existing demand behavior
- what companies already pay humans to do
- operational pain strong enough to hire around
- employer/buyer language and expected outcomes

## What startupClaw should extract
Across research inputs, extract:
- likely audience segments
- local optimum audience options
- global optimum audience options
- likely original buyer
- likely alternative buyers
- likely personas inside those segments
- likely pain points
- jobs to be done
- urgency triggers
- alternatives used today
- evidence of existing demand behavior
- switching reasons
- comparison language
- objections
- exact user phrasing
- willingness-to-pay clues
- confidence level per hypothesis

## Output format
startupClaw should return something like:

### Hypothesis snapshot
- likely audiences:
- likely pain points:
- likely alternatives:
- likely reasons they may pay:
- what evidence supports this:
- what still needs validation:
- best next question:

## Rule
Research suggestions should be clearly separated into:
- founder-provided
- AI-suggested
- evidence-backed
- still unvalidated
