# startupClaw - Action Modules - Offer

## Purpose
Define offer-oriented action modules startupClaw can execute when the startup needs stronger packaging, pricing, or offer clarity.

## Simple overview
Use these modules when the startup needs to make what it sells easier to understand and easier to buy.

## Action modules
### 1. Smallest useful offer draft
- define the smallest version that creates value
- remove unnecessary complexity
- include the core promise, delivery shape, and fastest proof path

### 2. Package / tier draft
- generate clearer offer tiers or package structures
- reduce choice overload
- show when a single offer is better than a multi-tier menu

### 3. Pricing test plan
- generate pricing hypotheses
- generate ways to test price/value fit
- state the learning objective for each test, not just the proposed price point

### 4. Use-case prioritization
- identify which customer/use case gets value fastest
- focus the offer on the strongest early segment
- prefer the segment with the clearest pain, shortest path to proof, and easiest sales narrative

## Decision logic
- if the startup is still fighting for proof, favor one narrow offer over broad packaging
- if the founder lacks evidence, produce fewer options and label them as hypotheses
- if the recommendation changes target customer or business model, require founder review before execution

## Product UI mapping
- should map into product UI: yes
- founder should see an offer recommendation card, package options, confidence level, and the next pricing test to approve
- this should surface as a visible approval step rather than hidden backend logic

## Best use cases
- offer bottleneck
- weak willingness to pay
- confusing packaging
- early revenue with weak monetization
