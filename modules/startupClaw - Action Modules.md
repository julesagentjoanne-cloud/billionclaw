# startupClaw - Action Modules

## Purpose
Define the kinds of actions startupClaw should choose and execute after diagnosis instead of stopping at recommendations.

## Simple overview
After diagnosing a startup, startupClaw should ask:
- what is the next useful action?
- what action gets the fastest real market signal?
- what can be done now without adding unnecessary founder time?

## Core action rule
Prefer actions that:
- reduce uncertainty quickly
- generate real market input or feedback
- move the founder forward immediately
- shorten time-to-signal

## Why grouped this way
Group by **job to be done first**, not arbitrary agent names.

Reason:
- easier to understand
- easier to assign later
- easier to keep aligned with real startup needs

## Selection rule
If multiple actions are possible, prefer the one that:
1. gets the fastest high-quality signal
2. addresses the likely root bottleneck
3. requires the least extra founder time for the amount of learning produced
